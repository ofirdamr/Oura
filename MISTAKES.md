### 2026-07-05 — Photos ingested before the Stage 2 pipeline existed never got embedded, and nothing backfilled them
**What:** The founder took a real selfie against the live `WED-2024` demo event and wasn't matched, despite appearing in nearly every one of its 17 photos. DB query confirmed all 17 photos were still `embed_status:'pending'` with zero `face_embeddings` rows — the pipeline had never run on them at all, so there was nothing for a selfie to match against.
**Why:** Face-embedding only triggers via an enqueue-on-upload call inside `POST /events/:event_id/photos` (added when Stage 2 shipped). These 17 photos were inserted directly into the DB during Stage 1 seeding, before that enqueue code existed — so they were silently skipped, and no backfill job was ever built to catch up already-existing photos once the pipeline went live.
**Correct approach:** Added `POST /admin/backfill-embeddings` (bearer-gated by a dedicated `ADMIN_BACKFILL_TOKEN` secret, not photographer auth) that re-enqueues any photo not yet `embed_status:'done'` into the existing `face-embed-queue` — reuses the already-correct queue consumer rather than duplicating embed/cluster logic. Any future manually-seeded or bulk-imported event needs this same backfill step run once. Worth checking `embed_status` distribution after any non-standard photo ingestion path, not just assuming the queue caught everything.

### 2026-07-05 — `embed()` had no request timeout, so one stalled Cloud Run response could hang a queue message for 15+ minutes
**What:** While running the backfill above, a batch of photos got stuck at `embed_status:'processing'` for 15+ minutes with zero movement — no error, no retry, nothing in live `wrangler tail` output. The Cloud Run service's own `/health` responded in 0.4s throughout, ruling out a wholesale outage.
**Why:** `apps/api/src/pipeline/embedClient.ts`'s `fetch()` call to the embedding service had no `AbortController`/timeout, so a single stalled response (connected but never returning, for reasons not fully root-caused — not image size, not R2 access) blocked the entire queue consumer invocation indefinitely instead of failing and letting Cloudflare Queues' retry/DLQ logic take over.
**Correct approach:** Added a 25s `AbortController` timeout around the `fetch()` in `embedClient.ts`, so a stalled request now fails fast (`embed_service_timeout`) and lets the existing retry/DLQ path work as designed. Confirmed live: after this fix, subsequently-stuck photos moved to a terminal `failed`/DLQ state within the timeout window instead of hanging. One photo (`c116b679-...`) still hung past the fetch timeout on retry — the stall isn't only in the Cloud Run round-trip, so if this recurs, check for missing timeouts on the R2 `.get()` call and the Supabase `match_faces`/insert calls too, not just the embed fetch.

### 2026-07-04 — Bidi space-collapse around an inline-styled number in Hebrew copy
**What:** On the Personal Gallery port, "מצאנו 12 תמונות" rendered as "מצאנו12תמונות" — both spaces around the `<span>`-wrapped `12` vanished — even though the JSX source had the spaces written correctly on both sides.
**Why:** Wrapping a number in its own inline element inside RTL text creates a bidi run boundary; the browser's bidi resolution collapsed the adjacent neutral space characters at that boundary. Plain numbers with no wrapping element (e.g. the "842" a few words later, not in a span) were unaffected.
**Correct approach:** Add `unicode-bidi: isolate` to any inline element that wraps a number/stat sitting inside Hebrew sentence text. Caught by visually verifying a cropped screenshot rather than trusting the JSX source diff — worth doing on every screen with inline counts.

### 2026-07-04 — Injected credential env vars carry a leading space
**What:** The provided `SUPABASE_URL` / `CLOUDFLARE_*` / R2 env vars each have a leading space in their value (`echo "[$SUPABASE_URL]"` prints `[ https://...]`). Used raw, this mangles curl URLs/headers and would push a malformed value into a Wrangler secret (a JWT/URL with a leading space breaks auth).
**Why:** The injection mechanism prepends a space; easy to miss because most output looks fine at a glance.
**Correct approach:** Trim before use, every time. In-shell (no external process, no added newline): `V="${VAR#"${VAR%%[![:space:]]*}"}"`, or `sed 's/^[[:space:]]*//'` for one-offs. Pipe secrets to `wrangler secret put` with `printf %s "$V"` (no trailing newline) and never echo the value. Applies to any future session using these creds.

### 2026-07-04 — `SUPABASE_URL` for supabase-js must be the base URL, not `.../rest/v1/`
**What:** The Worker's `GET /gallery/:token` (its first real DB call — it was a 501 stub before) returned `lookup_failed` (500) on every query. The `SUPABASE_URL` Wrangler secret had been set to a value ending in `/rest/v1/` (the injected container env var `SUPABASE_URL` here is `https://<ref>.supabase.co/rest/v1/`, not the bare project URL). supabase-js appends its own `/rest/v1/`, so requests hit `.../rest/v1//rest/v1/<table>` → PGRST125 "Invalid path" / 404.
**Why:** Two traps stacked: (1) the provided env var already has the PostgREST path baked in — surprising, and different from what `createClient` wants; (2) it stayed dormant because the earlier pass only tested PostgREST via direct `curl` (which correctly uses the `/rest/v1/` base) and never exercised the Worker's own Supabase client behind the stub. The health check only reports the secret is *present* (boolean), not that it *works*.
**Correct approach:** `createClient(url, key)` wants the PROJECT BASE URL (`https://<ref>.supabase.co`) — strip any trailing `/rest/v1/`. When wiring a Worker's DB client, exercise one real query end-to-end, not just a presence/health check; a "binding present" green light hides a malformed value. For direct `curl` to PostgREST the base is the opposite (`.../rest/v1/<table>`) — don't reuse the same string for both.

### 2026-07-04 — Signed numbers still bidi-reverse inside `unicode-bidi: isolate` in RTL context
**What:** On the Photo Editor's slider value readouts, a negative value rendered as `40-` instead of `-40`, even with `unicode-bidi: isolate` applied on the span (the fix documented in the earlier "bidi space-collapse" entry above).
**Why:** `unicode-bidi: isolate` creates a new bidi run, but that run's *base direction* still resolves from the element's own `direction` property (inherited `rtl` from the page here), not from the content. A plain positive number ("12") looks fine either way because pure digit runs (Unicode bidi type EN) aren't reordered relative to a base direction. But a leading minus sign is a weak/neutral character whose position relative to the digits *is* resolved according to the isolate's base direction - in an RTL-base isolate it gets placed on the trailing (right) side instead of the leading (left) side.
**Correct approach:** For any inline-isolated numeric value that can be negative (or otherwise signed/formatted, e.g. contains `-`, `+`, `%` in a specific order), set `direction: ltr` explicitly alongside `unicode-bidi: isolate` on that span - don't rely on isolate alone. Plain unsigned integers (the gallery photo-count case) don't need this, but anything with a sign does. Verify by actually driving the value negative in a screenshot, not just checking the default/positive case.

### 2026-07-04 — Toggle switch on/off direction inverted for RTL (Tailwind logical `start-*`/`end-*` positioning)
**What:** Built a custom toggle switch (auto-optimize, Photo Editor) using an absolutely-positioned thumb with Tailwind's logical `start-*` utility for the two states. Applies to any future custom switch/slider-thumb/handle component, not just this one.

### 2026-07-04 — A launched Playwright browser doesn't route external requests through this sandbox's proxy the way curl/Node do
**What:** Verifying the guest-flow wiring, a real click-through against the live Worker hung indefinitely on the very first `fetch` from the browser (`POST /events/:id/guests`) - no error, no response, just a permanent pending state. `curl`/Node requests to the same host worked fine throughout the session.

Prefer (b) for pure frontend-logic verification; only reach for (a) when the live round-trip itself is what needs proving (and that's usually better done with `curl`, which already works, rather than through a browser).

### 2026-07-04 — Mixing a logical inset (`start-1/2`) with a physical transform (`-translate-x-1/2`) mis-centers under RTL
**What:** The Gift Box Reveal's absolutely-positioned instruction pill was centered with `start-1/2 -translate-x-1/2`. Under the app's `dir=rtl`, this pushed the pill off toward the inline-start edge instead of centering it, and since its canvas wrapper is `overflow-hidden`, the pill got clipped to a couple of characters ("גר") on mobile. Looked fine on desktop only because the wrapper was wide enough to not clip the misplacement.
**Why:** `start-1/2` resolves to `inset-inline-start: 50%`, which in RTL maps to `right: 50%`. But `-translate-x-1/2` is a *physical* X translate (always toward physical-left). The two don't compose into a centered element the way `left-1/2 -translate-x-1/2` does in LTR — the logical inset and the physical translate pull in directions that only cancel correctly in LTR. Horizontal *centering* is actually direction-agnostic, so reaching for the logical `start-*` version of the LTR centering idiom is the trap: it silently breaks the thing it looks like it should fix.
**Correct approach:** For horizontally centering an absolutely-positioned element, don't translate at all — use `inset-x-0 mx-auto` (optionally `w-max`/`max-w-*`). `inset-x-0` is symmetric (`left:0;right:0`) so it's RTL-safe with no physical/logical mismatch, and `mx-auto` centers the fixed-width child. Reserve the `left-1/2 + -translate-x-1/2` idiom for cases where you genuinely can't set both insets, and if you must, keep both the inset and the translate on the *same* axis convention (both physical). Verify centering at a narrow (mobile) width where clipping actually surfaces, not just desktop.

### 2026-07-04 — Three.js/WebGL in Next App Router: SSR, disposal, and r185 API drift (first 3D screen)
**What:** Porting the Gift Box Reveal (the only non-static MVP screen) meant the first real Three.js + GSAP integration in this Next 16 / React 19 app. Several distinct footguns showed up that any future canvas/WebGL/3D screen will hit too.
**Why / what bit:** (1) **SSR** — `WebGLRenderer`, `document.createElement('canvas')`, `ResizeObserver`, `navigator.vibrate` etc. don't exist server-side; importing/rendering the scene during SSR would crash or warn. (2) **Lighting units** — Three r155+ dropped `useLegacyLights`; lights are physically based by default, so the mockup's `PointLight(color, 1.5)` renders almost black at the mockup's light distances. (3) **Deprecation noise** — `THREE.Clock` is deprecated in r185 and prints a console warning every session, which trips the "no console warnings" verification bar. (4) **GPU/memory** — without explicit teardown, a client-navigated-away 3D route leaks the WebGL context (browsers cap live contexts ~16, so repeated visits eventually fail to create one).
**Correct approach (reusable recipe for any future 3D/canvas screen):**
- Put the scene in its own `"use client"` component and `dynamic(() => import(...), { ssr: false, loading: <spinner/> })` it from the page. Do all WebGL/`window`/`document` work inside `useEffect`, never at module scope.
- For predictable lighting that matches a Stitch mockup's simple values, set `decay: 0` on point lights (opts out of physical inverse-square falloff) rather than blindly reusing the mockup's tiny intensities; tune against real rendered screenshots.
- Use `performance.now()` (or `THREE.Timer`) for elapsed time, not the deprecated `THREE.Clock`, so the console stays clean.
- Always return a teardown from the effect: `cancelAnimationFrame`, `gsap.killTweensOf(...)` every animated target, remove all pointer listeners + `ResizeObserver.disconnect()`, `.dispose()` every geometry/material/texture, then `renderer.dispose()` + `renderer.forceContextLoss()` and remove the canvas from the DOM.
- Verify with Playwright launched with `--use-gl=swiftshader` (software WebGL works headless), assert a real `getContext('webgl2')` is non-null, and screenshot at multiple animation points (closed / mid-open / revealed) — a 200 on the route proves nothing about whether the scene actually rendered. Note that Playwright's own `.screenshot()` emits `GL Driver Message ... ReadPixels` stalls under swiftshader; those are the screenshotter, not your code — filter them out of the "warnings" count.

### 2026-07-05 — `text-end` was used throughout for "normal right-aligned Hebrew text" - it means the opposite
**What:** Multi-line Hebrew paragraphs/labels across 11 files (34 occurrences) rendered flush-left with a ragged right edge - the mirror image of correct Hebrew body text, which should be flush-right/ragged-left (the same way English paragraphs are flush-left/ragged-right). Single-line short labels never exposed this since text-align has no visible effect when a block's width matches its content exactly - only genuinely wrapping, multi-line text (paragraphs, descriptions) revealed it, which is exactly why it kept surviving review.
**Why:** `text-align: end` is a logical value relative to reading direction: for RTL, "end" is the *far* side from where text begins - i.e. **left**, not right. The natural/idiomatic alignment for body text is always to the *start* of reading direction (left for LTR, right for RTL) - `text-align: start`, not `end`. `text-end` was used everywhere on the (incorrect) assumption that "end" meant "the far/right side for Hebrew." Verified empirically (not just reasoned about): measured per-line `getClientRects()` via a `Range` on the text node, both in an isolated `dir=rtl` test page and in the live component - `text-align:end` consistently produced flush-left/ragged-right; switching to `text-align:start` produced flush-right/ragged-left in the same measurement.
**Correct approach:** For normal Hebrew paragraph/label text that should read naturally right-aligned, use `text-start`, never `text-end`. Reserve `text-end`/`me-*`/`pe-*` for cases where content genuinely belongs at the *far* side from reading start (e.g., a trailing icon, a secondary value deliberately placed opposite the label) - the same "not universally wrong, check the actual intent" caveat that applied to `flex-row-reverse` applies here too, but for body text specifically, `text-start` is almost always what "looks right" for Hebrew means. When in doubt, don't trust visual review of short single-line text (it can't show the bug) - measure a genuinely wrapping block with `getClientRects()`/a `Range`, or check against the actual design screenshot.

### 2026-07-05 — A fixed per-event storage key + a shared "cache forever" media route silently broke logo re-uploads
**What:** The photographer reported uploading a new studio logo "did nothing" - the old logo stayed on screen with no visible error. The upload was actually succeeding end-to-end (R2 write, DB update, 200 response) - the bug was downstream: `POST /events/:event_id/branding/logo` stored every logo under a **fixed** key (`events/<event_id>/branding/logo.<ext>`), by explicit original design ("a studio has exactly one current logo"). But the shared `GET /media/*` route serves every R2 key with `Cache-Control: public, max-age=31536000, immutable`, justified by a comment claiming "content-addressed keys are unique per upload" - true for photos (`.../orig/<uuid>.<ext>`), false for the logo route, which reused the same URL every time. So a re-upload changed the bytes in R2 but not the URL, and the browser (plus any CDN edge cache) kept serving the year-old cached response against that unchanged URL - invisible in a code review, since the upload's own success path looked completely correct.
**Why:** "Fixed filename, intentionally overwrite on re-upload" and "cache this route forever, keys are unique" were each reasonable in isolation (in fact each had its own explanatory comment) but were never checked against each other - the fixed-key route silently violated the immutable-cache route's actual precondition. A bug that spans two files/routes each written correctly for their own stated assumption is easy to miss because neither file looks wrong on its own.
**Correct approach:** Before applying a shared "cache forever" policy to any route that serves content by key, audit every writer of that key space for whether the key is genuinely content-addressed (never reused for different bytes) or can be legitimately overwritten - a single mutable-key writer breaks the caching contract for everyone behind that route. Fixed here by making the logo key unique per upload too (`events/<event_id>/branding/logo-<uuid>.<ext>`, mirroring how photos already do it), so a re-upload naturally gets a fresh, cache-safe URL, and best-effort deleting the previous logo object from R2 afterward since only one is ever current. When a user reports "nothing happened" on an upload/save action, don't stop at confirming the write succeeded server-side - check whether the URL/key the client re-reads from actually changed.
