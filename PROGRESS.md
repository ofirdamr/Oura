Researched 3 credible, current sources: GitHub's own Spec-Kit (90k+ stars, constitution→specify→plan→tasks→implement), VoltAgent/awesome-agent-skills (official first-party skills from Cloudflare/Supabase/Vercel/Stripe/Google Labs), and google-labs-code/stitch-skills' "stitch-loop" (autonomous Stitch MCP design-generation loop).
- **Spec-Kit: declined, no changes.** It produces a new spec.md+plan.md+tasks.md+data-model doc per feature and explicitly makes no mention of token/context efficiency — heavier than what we have. Our existing shape already covers the same ground leaner: CLAUDE.md = its "constitution", `planner` agent = its specify/plan, `pm-orchestrator` = its tasks. Validates our structure rather than replacing it.
- **Official platform skills (Cloudflare Workers, Supabase Postgres, Next.js, Stripe): adopted as pointers, not imports.** Copying their content in would duplicate and go stale as platform APIs drift (confirmed — one guessed raw path 404'd). Instead added one-line WebFetch pointers to `backend-architect.md` (Cloudflare Workers + Supabase Postgres now, Stripe noted for when payments work starts) and `frontend-rtl.md` (Next.js App Router) — near-zero token cost, always-current.
- **stitch-loop: explicitly declined, flagged, not silently adopted or silently skipped.** It has the agent autonomously call Stitch MCP to generate new screens in a loop with no human in the middle — this directly conflicts with CLAUDE.md's locked rule ("Never design new visuals directly... founder runs Stitch himself"). Correctly named as a locked-boundary conflict rather than reinterpreted as fitting.
