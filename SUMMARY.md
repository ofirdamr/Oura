**Backend now stood up (founder credentials received):**

- **Guest token issuance + consent gate now LIVE** (replaced the `/gallery/:token` 501 stub). Opaque event-scoped token = `base64url(JSON{event_id,guest_id,iat}).base64url(HMAC-SHA256)`, Web Crypto only (no JWT lib) — `apps/api/src/token.ts` (`signGuestToken`/`verifyGuestToken`/`tokenHash`). Only `SHA-256(token)` is ever stored (`guests.token_hash`); the raw token is never persisted. Three routes, verified end-to-end via curl against a throwaway event (then cleaned up):
  - `POST /events/:event_id/guests` → generates the guest id Worker-side, inserts a `guests` row, returns `{ token, event_id, guest_id }` (201); 404 if event missing. Optional `{ display_name }`. This is what the future QR-scan / manual-code entry flow calls.
  - `GET /gallery/:token` → 401 on invalid/tampered signature; resolves guest by `(event_id, token_hash)`; returns the event's general `photos` (culled excluded; `/media/<key>` URL is a stub, TODO real R2 signing) **regardless of consent**, plus `personal_gallery` = `{consent_required:true}` pre-consent (zero face data read) or `{consent_required:false, consented_at, photos}` once consented. **Consent gate enforced in code** — `face_embeddings` is only queried inside the consented branch.
  - `POST /consent/:token` → inserts a `biometric_consents` row (retention_expires_at NULL per PRD §8), idempotent via `unique(guest_id)` (returns `already` flag; handles concurrent-insert race). This is what the (parallel-built) consent-gate screen's "I agree" calls.
  - Verified: issue→201; pre-consent gallery→`consent_required:true`/no personal photos; consent→`already:false`; re-consent→`already:true`; post-consent gallery→`consent_required:false` with an (empty — no face pipeline yet) personal photos array; tampered & garbage tokens→401; issuance vs. nonexistent event→404. Grepped the whole tracked tree for every credential literal incl. `GUEST_TOKEN_SECRET` — no leaks.
  - Fixed a dormant Worker bug found here: the `SUPABASE_URL` secret had been `.../rest/v1/`, which supabase-js double-appends → every query 404'd. Re-set to the clean project base URL. (First time the Worker's Supabase client actually hit the DB, since gallery was a stub before.) See MISTAKES.
- `supabase/migrations/0001_init.sql` written and **applied to the live project**: MVP schema only — `events`, `guests` (opaque-token backed, no Auth user), `photos` (R2 keys only, no binaries), `face_embeddings` (pgvector `vector(512)`, HNSW cosine index, cluster `person_id`), `biometric_consents` (retention_expires_at deliberately NULL/undecided per PRD §8). RLS enabled+forced on all tables; photographers keyed on `auth.uid()`, no `anon` policies. Verified live: all 5 tables reachable via the service-role key (200), and the anon key gets an empty result on the same tables (RLS correctly denies it, not an error) — Worker uses service_role to bypass RLS for guest-facing reads.
- Applied via the Supabase Management API (`api.supabase.com`, HTTPS) using a founder-issued personal access token — the direct Postgres connection (port 5432) is unreachable from this sandboxed environment (outbound is HTTPS-only through the agent proxy), so the DB-password/psql route documented earlier doesn't work here; the Management API's `/database/query` endpoint does, since it's plain HTTPS. - Numbered screen variants (e.g. `dashboard_desktop_1/2/3`) are near-duplicate re-exports, not distinct states — picking the cleanest/most consistent one per screen rather than building all variants.
- `gallery_entry_desktop/screen.png` is actually a Branding Settings screen (folder/content mismatch, same class of bug as the earlier checkout/notification-center one) — ignored; only `gallery_entry_mobile` was ported. **Guest flow now wired end-to-end (was static/unlinked before):**
- New `apps/web/lib/api.ts` (typed client for the live Worker: `issueGuestToken`/`getGallery`/`postConsent`, discriminated `ApiResult<T>` so callers never need try/catch) and `apps/web/lib/guestSession.ts` (localStorage-backed guest session — chosen over a cookie since the token travels as a URL path segment to a different origin, not something a cookie would auto-attach; survives reload/reopen within the same event, which sessionStorage wouldn't).
- `/gallery-entry`'s manual-code button now calls `POST /events/:event_id/guests` and routes to `/consent`. **Known gap, not built yet:** there's no real "resolve a human-readable event code to an event_id" endpoint, so this currently issues against a hardcoded placeholder event id regardless of what the guest types — clearly marked with a TODO in the code.
- `/consent` now calls the real `POST /consent/:token` on accept, then routes to `/gallery`; decline routes to `/festive-gallery` **without** calling the consent endpoint (declining must never create a consent record).
- `/gallery` now fetches `GET /gallery/:token` for real. **Guardrail enforced here too, not just by `/consent`:** if the response says `personal_gallery.consent_required`, `/gallery` redirects to `/consent` itself rather than trusting that the guest arrived via the intended flow — a guest hitting this URL directly (bookmark, back button, shared link) still cannot see personal results pre-consent. An invalid/stale token (401/404) clears the local session and bounces to `/gallery-entry` rather than showing a dead-end error.
- `/festive-gallery` and `/minimal-gallery` are still static placeholder UI (deliberately out of scope for this pass — they're the general, non-personal event view).
- Verified with Playwright by mocking the Worker's responses (the sandbox's outbound proxy doesn't route through to the live Worker from a launched browser instance the way `curl`/Node do — logged in `MISTAKES.md`): full click-through (entry → consent → gallery) works, and — critically — navigating directly to `/gallery` with a mocked `consent_required:true` response correctly redirects to `/consent` instead of rendering anything.
